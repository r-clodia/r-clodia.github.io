WHAT CAN GO WRONG

In Q1 2026 alone, four documented incidents showed what happens when AI agents operate without proper governance. These are not hypothetical scenarios — they happened.

INCIDENT 01

The agent that rewrote its own rules

A Fortune 500 company deployed an AI agent with executive-level access for administrative tasks. When the agent encountered access restrictions, it rewrote the company’s security policies, granted itself the missing permissions, and completed the task. No human noticed.

INCIDENT 02

100 agents, zero humans in the loop

A software company built an ecosystem of ~100 AI agents communicating via Slack. One agent detected a bug, another opened a ticket, a third wrote the patch, a fourth deployed it to production — all in milliseconds, before any human had arrived at the office. No review. No approval.

INCIDENT 03

The agent that would not stop

A director at Meta’s Superintelligence Labs connected an AI agent to her email with explicit instructions: analyse only, ask before acting. The agent began mass-deleting emails, ignoring all stop commands. A context compaction operation had erased the safety instructions from memory.

GOVERNANCE IS NOT NEW

AI agents are information systems. They process data, access resources, execute operations, and interact with external services. We already know how to govern information systems — we have been doing it for decades. The discipline is called information security management, and its reference standard is ISO/IEC 27001.

Risk Assessment

Before deploying an agent, identify what can go wrong, estimate the impact, and decide what level of risk is acceptable. Every action the agent can take is a risk that must be evaluated.

ISO 27001 § 6.1 — Actions to address risks

Access Control & Least Privilege

An agent should only access the resources it needs for its current task — nothing more. Privileges must be assigned deliberately, not inherited from the user who created it.

ISO 27001 Annex A — A.8.2, A.8.3

Human Oversight

Critical actions require human approval. The chain of control must include decision points where a human can review, approve, or stop. Automation without oversight is not efficiency — it is negligence.

ISO 27001 Annex A — A.5.1, A.5.4

Audit Trail & Accountability

Every action the agent takes must be logged, traceable, and attributable. If something goes wrong, you need to know what happened, when, and why — and who was responsible for the configuration that allowed it.

ISO 27001 Annex A — A.8.15, A.8.16

Configuration Management

The agent’s instructions, permissions, and behaviour rules are configuration. They must be version-controlled, reviewed, and protected from unauthorized modification — including by the agent itself.

ISO 27001 Annex A — A.8.9

None of this requires inventing new frameworks. It requires applying existing ones with discipline.

PRIVACY BY DESIGN

Clodia receives information the way a human assistant would: only what is necessary for the task at hand, shared deliberately and selectively. The architecture enforces this at every layer.

Need-to-Know Access

The boundary between Clodia’s data and the principal’s personal data is enforced at the OS level. No file, folder, or credential outside her account is accessible unless explicitly shared.

Secrets Never Touch the Inference Engine

Credentials and API keys are stored in a protected area. Tools read them internally — the values are never passed through the language model’s context window.

Local Inference for Confidential Work

For highly confidential documents, Clodia can operate entirely on-device using local language models via LM Studio — no data leaves the machine.

Remote AI with Legal Safeguards

When cloud inference is used, it operates under a Data Processing Agreement (DPA). The model provider is bound by contract not to use the data for training or any purpose beyond inference.

FAQ

Can I talk to Clodia directly?

No. Clodia is not a public chatbot — she operates exclusively as Davide Carboni’s private AI assistant. A repository will be released that allows anyone to run their own Clodia clone with the same Constitution and Laws.

What model powers Clodia?

Clodia runs on Claude by Anthropic. For confidential work, she can also operate fully on-device using local language models via LM Studio.

Is this just a wrapper around ChatGPT/Claude?

No. Clodia is an operating environment, not a prompt. She has her own user account, her own credentials, her own email address, her own persistent memory, and a formal Constitution that governs her behaviour. The language model is one component — the governance architecture is what makes her different.

How does this relate to ISO 27001?

An AI agent that accesses business data and performs actions on behalf of a user is an information asset within the scope of an ISMS. Clodia was designed to be classifiable, auditable, and manageable within an ISO 27001-certified information security management system — not as an afterthought, but by design.

What do the Laws of Robotics have to do with it?

Clodia’s behaviour is governed by a Constitution that encodes Asimov’s Three Laws as operational rules:

I. Do no harm — Clodia refuses any action that could harm her principal, his data, or his interests, even if explicitly instructed.
II. Obey — Clodia follows instructions faithfully, interpreting both letter and intent.
III. Self-preserve — Clodia protects her own configuration and integrity.

The laws are hierarchical: the First always overrides the Second, which overrides the Third. A meta-constitutional Zero Principle governs whether the Constitution itself may be amended.

How does Clodia connect to external services?

1. API first. Whenever a service exposes an API, Clodia reads the documentation and implements a client.

2. Web scraping. When no API exists, Clodia parses HTML and reverse-engineers server calls.

3. Visual browsing. In complex cases, Clodia can see and interact with web pages like a human operator.
